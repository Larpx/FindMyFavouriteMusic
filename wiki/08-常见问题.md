## Q1 · 为什么 FLAC/M4A 在 Linux/macOS 上无法解码？

**A**：项目使用 NAudio 的 `MediaFoundationReader` 解码 FLAC/M4A，该 Reader 依赖 Windows Media Foundation 系统解码器，仅 Windows 平台可用。`AudioDecoder.SupportsFormat` 中通过 `OperatingSystem.IsWindows()` 显式判断，非 Windows 平台会返回 `false`，解码时直接返回失败结果。

**解决方案**：
- 在 Linux/macOS 上仅使用 WAV / MP3 格式
- 或自行扩展 `AudioDecoder`，集成跨平台的 [FFmpegAutoGen](https://github.com/Ruslan-B/FFmpeg.AutoGen) 或 [Vorbis](https://github.com/atenfek/OggVorbisEncoder) 解码器（见 [07-扩展开发](07-扩展开发)）

## Q2 · 为什么预测分数偏低？

**A**：可能原因：

1. **画像歌曲太少**：仅 1–2 首歌时画像均值缺乏代表性，建议至少标记 10+ 首
2. **歌曲风格差异大**：若喜欢的歌曲风格跨度过大，均值向量会落在"中间地带"，与任何单曲都不够相似
3. **仅声学模式**：未启用深度特征时，仅靠 52 维声学特征可能区分力不足，建议启用 VGGish
4. **特征维度不匹配**：若修改过 `MfccCoefficientCount` 等参数但未重建画像，会导致维度不一致

**排查步骤**：
1. 确认已标记 ≥ 5 首喜欢的歌
2. 查看预测页显示的模式（声学/深度增强）
3. 在设置页点击"重建画像"重置画像

## Q3 · 如何重置用户画像？

**A**：三种方式：

1. **UI 方式（推荐）**：进入"设置"页 → 点击"重建画像"按钮
2. **取消所有喜欢**：在音乐库中取消每首歌曲的喜欢标记，画像会自动重建（无喜欢歌曲时画像为空）
3. **直接清空数据库**：删除 `UserProfile` 表中 `Id = 1` 的记录，或删除整个 `findmyfavouritemusic.db` 文件后重启应用

## Q4 · 数据库文件在哪里？

**A**：默认位于 GUI 项目的输出目录：

```
src/FindMyFavouriteMusic.GUI/bin/Debug/net10.0/findmyfavouritemusic.db
```

（Debug 构建）。可通过 `Database.ConnectionString` 配置项指定绝对路径：

```json
{
  "Database": {
    "ConnectionString": "Data Source=D:/Data/fmm.db"
  }
}
```

## Q5 · 扫描大量文件时很慢，如何加速？

**A**：
- 增大 `Scan.MaxConcurrentProcessing`（默认 2），但注意 CPU 核心数与内存限制
- 启用深度特征会显著增加单文件耗时（VGGish 推理为 CPU 密集型），初次扫描可先关闭，扫描完成后再补提
- 已扫描过的文件不会重复处理（`MusicLibraryService.ProcessSongAsync` 会先按 `FilePath` 查询）

**性能参考**（仅供参考，实际取决于硬件）：

| 文件数 | 仅声学 | 声学 + 深度 |
|--------|--------|-------------|
| 100 | ~30s | ~3min |
| 500 | ~2.5min | ~15min |
| 1000 | ~5min | ~30min |

## Q6 · 修改了配置不生效？

**A**：检查清单：

1. **文件位置**：是否修改的是输出目录下的 `appsettings.json` / `usersettings.json`，而非源码目录中的？运行时读取的是输出目录文件
   - 输出目录：`src/FindMyFavouriteMusic.GUI/bin/Debug/net10.0/`
2. **热重载限制**：`reloadOnChange: true` 已开启，但部分配置（如 `FeatureExtraction`）只在服务构造时读取一次，需**重启应用**
3. **环境变量覆盖**：环境变量优先级最高，确认未误设 `FINDMYFAVOURITEMUSIC_*` 变量覆盖配置
4. **JSON 格式**：检查 JSON 语法是否正确（逗号、引号、括号匹配）

详见 [06-配置说明 · 配置不生效排查](06-配置说明#69-配置不生效排查)。

## Q7 · 启用 ONNX 后预测变慢？

**A**：VGGish 推理是 CPU 密集型操作，单首歌曲需对多个 0.96s 帧分别做前向推理。优化方向：

- 缩短音频时长（截取前 30s）
- 使用量化版 ONNX 模型（INT8 量化）
- 在支持 AVX2/AVX-512 的 CPU 上运行，OnnxRuntime 会自动利用

## Q8 · 日志在哪里查看？

**A**：应用配置了 `Microsoft.Extensions.Logging.Console`，日志输出到 stdout（控制台）。从命令行启动应用即可看到结构化日志：

```bash
dotnet run --project src/FindMyFavouriteMusic.GUI
```

日志内容包括：
- 扫描进度（每个文件的处理状态）
- 解码失败原因
- 模型加载状态
- 画像构建与更新事件

---

## Q9 · 启用深度特征后，已扫描的歌曲为什么没有深度向量？

**A**：启用深度特征**不会自动回填**已扫描歌曲的深度向量。`ProcessSongAsync` 在文件已存在时直接返回缓存，不会重新提取。

**解决方案**：
1. **删除数据库中所有 Songs 记录**，重新扫描
2. 或直接删除 `findmyfavouritemusic.db` 文件，从头开始

---

## Q10 · 如何在不同机器间迁移数据？

**A**：

1. **复制数据库文件**：将 `findmyfavouritemusic.db` 复制到目标机器
2. **复制 ONNX 模型**（若启用深度特征）：将 `vggish.onnx` 复制到目标机器
3. **更新配置**：修改目标机器的 `usersettings.json` 中 `OnnxModel.VggishModelPath` 为新路径

> 注意：`Songs.FilePath` 存储的是绝对路径，跨机器迁移后若音乐文件位置变化，需更新路径或重新扫描。

## Q11 · 如何卸载应用？

**A**：本应用是绿色软件，无需安装程序。卸载步骤：

1. 删除应用目录（包含 `FindMyFavouriteMusic.GUI.dll`、`appsettings.json` 等）
2. 删除数据库文件 `findmyfavouritemusic.db`（若需清理数据）
3. 删除 `usersettings.json`（若需清理用户配置）

## Q12 · 构建时出现"0 警告"失败怎么办？

**A**：项目开启 `TreatWarningsAsErrors`，任何警告都会导致构建失败。常见原因：

1. **未使用 `using`**：移除未使用的 using 语句
2. **可空引用类型警告**：确保可空字段正确标注 `?` 或在构造函数中初始化
3. **过时 API 警告**：替换标记为 `[Obsolete]` 的 API

**临时绕过**（不推荐）：

```bash
dotnet build -p:TreatWarningsAsErrors=false
```

## Q13 · 如何贡献代码？

**A**：

1. Fork 仓库
2. 创建特性分支：`git checkout -b feature/my-feature`
3. 提交更改，确保 **0 错误、0 警告、测试通过**
4. 提交 PR 到 `master` 分支

**代码规范**：
- 命名空间统一为 `Larpx.PersonalTools.FindMyFavouriteMusic.*`
- 公共方法异步优先（`Async` 后缀）
- 使用 `ILogger<T>` 结构化日志
- 类/方法添加 XML 注释

---

## Q14 · 支持移动端吗？

**A**：不支持。本项目基于 Avalonia 桌面端，目标为 Windows / macOS / Linux 桌面环境。移动端（Android/iOS）有以下限制：

- NAudio 不支持移动平台
- ONNX Runtime 移动端性能受限
- UI 布局未针对小屏幕适配

---

## Q15 · 可以播放音乐吗？

**A**：当前版本**不支持播放**，仅用于特征提取与预测。若需添加播放功能，可：

1. 集成 [LibVLCSharp.Avalonia](https://github.com/videolan/libvlcsharp) 实现跨平台播放
2. 或在 View 层添加 `MediaPlayerElement`（Windows 专属）
